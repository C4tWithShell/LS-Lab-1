Sparse-file virtual disk

Then I moved on to the SPARSE file. To do this, I run the command dd if=/dev/zero of=Guest2.img bs=1 count=0 seek=15G. It will create a sparse-file with a count size of zero, which will be extended to the size 15G by using the seek option. Then I run several commands to check the file as shown in Figure 5.


Figure 5 - Creating the SPARSE-file

Then I used the command sudo losetup --show -f Guest2.img to set up the image as a disk. To mount it, I need to create a filesystem as shown in Figure 6.


Figure 6 - Creating ext4 filesystem

Here first I created the gpt partition table and primary ext2 partition. Then I created an ext4 filesystem on the primary partition. After that, I mounted a new partition to /mnt.

Now let’s create an image using debootstap - sudo debootstrap focal /mnt http://ru.archive.ubuntu.com/ubuntu. After successfully installing the base system, I run the command block:

// I mount the /dev tree inside the image to have access to chroot
sudo mount -o bind /dev /mnt/dev
// chroot to the image
LANG=C sudo chroot /mnt
// Mount proc and sys environment
mount -t proc proc /proc
mount -t sysfs sysfs /sys

Then I added an entry to the fstab file using the command

EOF
root@st12:/# cat << EOF > /etc/fstab
UUID=$(blkid /dev/loop27p1 | cut -d\" -f2) / ext4 errors=remount-ro 0 1
EOF

Also, I made sure that the updates and security repositories are enabled:

cat << EOF > /etc/apt/sources.list
deb http://ru.archive.ubuntu.com/ubuntu focal main restricted universe multiverse
deb http://ru.archive.ubuntu.com/ubuntu focal-updates main restricted universe multiverse
deb http://security.ubuntu.com/ubuntu focal-security main restricted universe multiverse
EOF

After that, I update it with apt update and apt -y dist-upgrade.

The next step will be to install the kernel image, the GRUB bootloader, SSH, and other tools - apt -y install linux-image-generic grub2-common openssh-server bridge-utils ethtool.

I changed some grub options and applied it using update-grub (I showed it later). Then I wrote grub to the disk grub-install /dev/loop27 --force.

// Creating admin group
groupadd admin
// Creating user account to log in with that is a member of this group
useradd -s /bin/bash -m -d /home/user -G admin user
// Setting password for the user
passwd user
// Exit chroot
exit
// unmount filesystem
sudo umount /mnt/{dev,proc,sys} /mnt
// Remove loopback device
sudo losetup -d /dev/loop27

So, now I have added the VM with the created img file (Figure 7).

virt-install --virt-type kvm --name Guest2 --vcpus 2 --memory 4000 --os-variant ubuntu20.04 --disk ~/Guest2.img --import --network bridge=br12 --graphics none

Task 3 - Cluster validation

  1. Take team member 1’s favorite guest (virtual disk and configuration) and put it in the shared storage
  2. Take team member 2’s favorite guest (virtual disk and configuration) and put it in the shared storage
  3. Eventually fix the paths in the configuration and validate that both guests run as well as before
  4. Now shut them down and run them on the other team member’s machine/host (cold-migration)
  5. Now don’t even shut them down while migrating… (hot-migration/live-migration)

First, I installed NFS and the necessary packages.

sudo apt update
sudo apt install nfs-kernel-server ssh-askpass

Then I changed my br12 interface to br0 to prevent configuration issues. Then we decided that the Polina workstation would provide shared storage. So she changed the permission to her folder - sudo chown nobody:nogroup /var/lib/libvirt/images/. And then added a new line to the etc/export file - /var/lib/libvirt/images *(rw,sync,no_subtree_check).

/var/lib/libvirt/img - directory that I will share with Polina
* - client (all)
rw - read-write permission to client
sync - forces NFS to write changes to disk before sending a response.
no_subtree_check - prevents checking the nested tree

Then she restarted nfs service - service nfs-kernel-server restart.

At this point, I mount the storage - mount 10.1.1.207:/var/lib/libvirt/images /var/lib/libvirt/images/. Then I changed the permission to my VM image and copied it to the storage. Figure 20 shows the shared storage.


Figure 20 - The shared storage

There are my image (Guest 1.qcow2) and one of Polina’s images in two formats qcow2 and img.

Then we tried cold migration. To do this, I created the file on my VM, and then shut it down ( Part 1 video from Polina). After that, I started Polina’s VM from shared storage ( Part 2 video) and opened this file with the message “Hello world” as shown in Figure 21.


Figure 21 - Cold-migration

The next step is hot-migration. To do this, we used the GUI. In the Part 3 video, Polina started her VM and initialized the migration process to my workstation. During this process, I ping the VM as shown in the Part 4 video.

At first, it took 0.5 ms. Then, when the migration starts, it increases to 1-2 ms, and when the migration is completed, it decreases to 0.2 ms, and the new VM is displayed in Virtual Manager (ubuntu20.04qow2).

The same process was done by me. I started my guest VM, then created a connection to the Polina workstation, as shown in Figure 22.


Figure 22 - Adding connection

After that, I started the migration with the same result - the virtual machine on my workstation was suspended, while it appears on Polina’s computer, where it running.
